# 要件定義（Requirements）

## 目的

子ども向け教育アプリの開発を、AI エージェントと品質ゲートを用いて安全かつ再現可能に進める。
当面はアプリ本体よりも、MVP パイプライン、要件管理、監査フロー、CI を先に整備し、継続的に機能を追加できる土台を作る。

## スコープ

### 対象（本リポジトリで扱う）

- 教育アプリ開発に必要な計画、要件、制約、運用手順の SSOT 管理
- Python による MVP データ処理パイプラインの実装
- 制約チェック、エラーハンドリング、再現性確保のための基盤整備
- CI、静的解析、テスト、自動監査の整備
- 将来のモバイルアプリ機能開発に備えた開発プロセスの標準化

### 非対象（本リポジトリでは行わない）

- 本番向け iOS / Android アプリ画面の完成実装
- 認証、課金、通知、保護者ダッシュボードなどの本番機能
- 実ユーザーデータの保存・分析
- 本番インフラの構築、監視、運用自動化の完成形

## 用語

| 用語           | 定義                                                         |
| -------------- | ------------------------------------------------------------ |
| 正本（SSOT）   | Single Source of Truth。docs/ 配下の文書を唯一の信頼源とする |
| 受入条件（AC） | Issue/PR が完了と判断できる基準                              |
| MVP パイプライン | 最小限の入力、制約評価、処理、出力までを一貫して実行できる最小構成 |
| 教育データ | 学習コンテンツ、評価値、進捗指標など、教育アプリで扱う対象データの総称 |
| 制約チェック | 入力値や設定値が安全かつ許容範囲内かを評価する処理 |

## 成功条件（ゴール）

- CI が品質ゲートとして機能し、失敗時はマージできない
- 仕様（requirements/policies/constraints/plan/ADR）が更新され、会話ログ依存が最小化される
- MVP パイプラインが設定ファイル経由でローカル実行できる
- 制約違反時に安全側で停止し、理由を追跡できる
- 教育アプリ本体の実装を始める前に、最小限のテスト、監査、運用フローが成立している

## 機能要件（FR）

### FR-001 MVP 教育データ処理パイプライン

- 入力：識別名と数値列を持つ教育データ入力、および任意の設定ファイル
- 出力：合計値、件数、平均値、実行状態、タイムスタンプを含む処理結果
- 失敗時：制約違反またはドメインエラーとして停止し、理由をログに記録する
- 担当モジュール：`src/my_package/core/`, `src/my_package/domain/`, `scripts/run_pipeline.py`

### FR-010 設定ファイルによる実行制御

- 入力：TOML 形式の設定ファイル
- 出力：最大入力件数、倍率、出力先などを含む実行時設定
- 担当モジュール：`src/my_package/core/config.py`, `configs/`

### FR-020 制約評価と安全停止

- 入力：MVP パイプラインへ渡される入力値列と実行設定
- 出力：制約を満たす場合のみ処理継続し、違反時は理由コード付きで停止する
- 失敗時：`ConstraintViolationError` を送出し、以降の処理を継続しない
- 担当モジュール：`src/my_package/domain/constraints.py`, `src/my_package/core/exceptions.py`

### FR-030 品質ゲート付き開発フロー

- 入力：ソースコード、テストコード、ドキュメントの変更
- 出力：lint、型チェック、テスト、policy check による検証結果
- 失敗時：CI で失敗として扱い、マージ不可とする
- 担当モジュール：`.github/workflows/`, `ci/policy_check.py`, `tests/`

## 非機能要件（NFR）

### NFR-001 再現性

- 設定、シード、データバージョン、コードバージョン（コミットSHA）を記録する
- 同一入力に対して同一出力が得られることをテストで担保する
- `configs/pipeline_default.toml` を基準設定として、ローカルで同一結果を再現できること

### NFR-010 セキュリティ

- 秘密情報をコミットしない
- CI の policy_check で機械的に検査する
- エージェント定義、CI 設定、スクリプトを含むリポジトリ全体で秘密情報混入を防ぐ

### NFR-020 保守性

- 重要ロジックは単体テストを必須とする
- 変更は 1PR で理解可能な粒度に分割する
- コードの識別子は英語、説明文・PR・Issue は日本語で記述する
- `ruff`, `mypy`, `pytest` が継続実行できる構成を維持する
- 主要モジュールのテストカバレッジ 80% 以上を目標とする

### NFR-030 運用容易性

- ローカルで最小コマンドで実行できる
- 失敗時の復旧手順を runbook に記載する
- `uv run python scripts/run_pipeline.py` で MVP パイプラインを起動できる
- 開発者が docs を見れば、次に着手すべきタスクが判断できる

## 受入条件（全PR共通）

| ID     | 条件                                                                    |
| ------ | ----------------------------------------------------------------------- |
| AC-001 | 変更は要件・ポリシー・計画のいずれかに整合する                          |
| AC-010 | 必要なテストが追加または更新されている                                  |
| AC-020 | CI（lint/type/test/policy_check）が成功している                         |
| AC-030 | 変更に応じて正本（docs）が更新されている                                |
| AC-040 | 検証手順と結果が PR 本文に記載されている                                |
| AC-050 | `docs/constraints.md` に定義された制約（現時点では `C-001`, `C-002`）と `docs/policies.md` に反する変更がない（例外は ADR に記載されている） |
